Git and GitHub - Complete Session Guide

A comprehensive guide for version control and collaboration

1. What is Git?

Git is a distributed version control system that tracks changes in your code over time. It allows multiple developers to work on the same project without overwriting each other's work, maintains a complete history of all changes, and enables you to revert to previous versions if needed.

Key Points:
  • Distributed system - every developer has a full copy of the repository
  • Tracks changes, not entire files (efficient storage)
  • Works locally, then syncs with remote repositories

2. Version Control Benefits

Version control solves critical problems in software development:

  • History tracking - See who changed what, when, and why
  • Collaboration - Multiple people can work simultaneously
  • Backup - Code is safely stored and recoverable
  • Experimentation - Try new features without breaking working code
  • Revert changes - Undo mistakes easily
  • Code review - Review changes before merging into main codebase

4. Basic Git Workflow

The typical flow of working with Git involves four stages:

  1. Working Directory - Your local files where you make changes
  2. Staging Area - Where you prepare changes to commit
  3. Local Repository - Your committed changes stored locally
  4. Remote Repository - The shared repository on GitHub

5. Essential Git Commands

Initial Setup

# Configure your identity (one-time setup)
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# Check configuration
git config --list

Starting a Repository

# Initialize a new Git repository
git init

# Clone an existing repository
git clone https://github.com/username/repository.git

Checking Status

# Check status of your files
git status

# See what changed in files
git diff

6. git add - Staging Changes

Moves changes from your working directory to the staging area. This is where you prepare what will be included in your next commit. 

# Stage a specific file
git add filename.txt

# Stage multiple files
git add file1.txt file2.txt

# Stage all changes in current directory
git add .

# Remove file from staging (unstage)
git reset filename.txt
Key Point: Staging allows you to selectively choose which changes to commit, rather than committing everything at once.

7. git commit - Saving Changes

Takes the staged changes and saves them to your local repository with a descriptive message. 

# Commit with message
git commit -m "Add user login feature"

# Stage and commit tracked files in one step
git commit -am "Fix bug in payment process"

# View commit history
git log

Best Practices for Commit Messages

  • Be specific and descriptive
  • Explain what and why, not how
Examples:
✅ Good: "Fix null pointer exception in user authentication"
❌ Bad: "Fixed stuff" or "Updates"

8. Checking Out Specific Commits

Git allows you to travel back in time to view or work with previous versions of your code. You can checkout a specific commit to see what your project looked like at that point in history.

Why checkout specific commits?

  • Investigate when a bug was introduced
  • Review the state of code at a specific point in time
  • Recover lost code from a previous version
  • Create a new branch from a historical point

Finding Commits

# View commit history with full details
git log

Checking Out a Specific Commit

# Checkout using commit hash (first 7 characters is enough)
git checkout abc1234

# Checkout using full commit hash
git checkout abc1234567890abcdef1234567890abcdef12345

# View information about a specific commit
git show abc1234
Important: When you checkout a specific commit, you enter a "detached HEAD" state. This means you're not on any branch. Any new commits you make won't belong to any branch and might be lost.

9. Branches

Branches allow you to create separate lines of development. Think of it as creating a copy of your code where you can experiment without affecting the main codebase.

Why use branches?

  • Work on new features without breaking working code
  • Multiple developers can work on different features simultaneously
  • Keep main branch stable and deployable

Creating Branches

# Create a new branch
git branch feature-login

# Create and switch to new branch
git checkout -b feature-login

# or (newer syntax)
git switch -c feature-login

# List all branches
git branch

10. Switching Branches

Move between different branches to work on different features or versions. 

# Switch to existing branch
git checkout feature-login

# or (newer syntax)
git switch feature-login

# Switch to main/master branch
git checkout main
Important: Make sure to commit or stash your changes before switching branches, or Git will warn you.

11. Merging Branches

Combine changes from one branch into another.

# Switch to branch you want to merge INTO
git checkout main

# Merge feature branch into current branch
git merge feature-login

# Delete branch after merging
git branch -d feature-login

12. What are Merge Conflicts?

Conflicts occur when Git can't automatically merge changes because two people modified the same part of the same file differently.

Example Scenario

  • Developer A changes line 10 in app.js
  • Developer B also changes line 10 in app.js
  • Git doesn't know which change to keep

Conflict Markers

<<<<<<< HEAD
const name = "Alice";
=======
const name = "Bob";
>>>>>>> feature-branch

Resolving Conflicts

Step 1: Git will mark the conflicted files 
git status
Step 2: Open the file and manually resolve conflicts
Remove conflict markers and choose/combine changes
Step 3: Stage the resolved file 
git add filename.js
Step 4: Complete the merge 
git commit -m "Resolve merge conflict in filename.js"

Tips to Avoid Conflicts

  • Pull latest changes frequently
  • Communicate with team about which files you're working on
  • Keep commits small and focused
  • Merge main branch into your feature branch regularly

13. Working with Remote Repositories 

# View remote repositories
git remote -v

# Add a remote repository
git remote add origin https://github.com/username/repo.git

# Fetch changes from remote (doesn't merge)
git fetch origin

# Pull changes from remote (fetch + merge)
git pull origin main

# Push your commits to remote
git push origin feature-login

# Push and set upstream tracking
git push -u origin feature-login

# After setting upstream, simply use
git push

14. What is a Pull Request (PR)?

A Pull Request is a GitHub feature (not Git itself) that lets you tell others about changes you've pushed to a repository. It's a request to merge your branch into another branch (usually main).

Purpose

  • Code review - Team members can review your changes
  • Discussion - Comment on specific lines of code
  • Quality control - Ensure code meets standards before merging
  • Documentation - Creates a record of what changed and why
Important: PR is NOT the same as git pull - common confusion!

15. Creating a Pull Request

Step 1: Push your branch to GitHub 
git push origin feature-login
Step 2: On GitHub website
  • Navigate to the repository
  • Click "Pull requests" tab
  • Click "New pull request"
  • Select your branch to merge from
  • Select target branch (usually main)
  • Review the changes shown
Step 3: Fill in PR details
  • Title: Clear, descriptive summary
  • Description: Explain what changed, why, and any special considerations
  • Add reviewers (team members who should review)
  • Add labels if your team uses them
Step 4: Create the PR
Click "Create pull request"
Step 5: Address feedback
  • Make requested changes locally
  • Commit and push again
  • PR automatically updates with new commits
Step 6: Merge
  • Once approved, click "Merge pull request"
  • Delete the branch after merging

16. Collaboration Workflow

How teams work together on the same project:

Typical Team Workflow

1. Clone the repository 
git clone https://github.com/team/project.git
cd project
2. Create a feature branch 
git checkout -b feature-user-profile
3. Make changes and commit 
git add .
git commit -m "Add user profile page"
4. Keep your branch updated 
git checkout main
git pull origin main
git checkout feature-user-profile
git merge main
# Resolve any conflicts
5. Push your branch 
git push origin feature-user-profile
6. Create Pull Request on GitHub
7. Team reviews and discusses
8. Make changes if requested 
# Make changes
git add .
git commit -m "Address review comments"
git push origin feature-user-profile
9. Merge after approval
10. Update local main 
git checkout main
git pull origin main
git branch -d feature-user-profile

17. Collaboration Best Practices

  • Pull before you push - Always get latest changes first
  • Commit often - Small, focused commits are easier to review
  • Write clear commit messages - Your future self will thank you
  • Never commit directly to main - Always use feature branches
  • Keep PRs small - Easier to review and less likely to have conflicts
  • Review others' code - Helps catch bugs and share knowledge
  • Communicate - Let team know what you're working on

18. Common Git Commands Cheat Sheet

Status and Information

git status                    # Check status
git log                      # View commit history
git log --oneline            # Compact history
git diff                     # See unstaged changes
git diff --staged            # See staged changes

Basic Workflow

git add <file>               # Stage file
git add .                    # Stage all changes
git commit -m "message"      # Commit changes
git push                     # Push to remote
git pull                     # Pull from remote

Branching

git branch                   # List branches
git branch <name>            # Create branch
git checkout <branch>        # Switch branch
git checkout -b <branch>     # Create and switch
git merge <branch>           # Merge branch
git branch -d <branch>       # Delete branch

Undo Changes

git checkout -- <file>       # Discard changes in file
git reset HEAD <file>        # Unstage file
git revert <commit>          # Undo a commit (safe)
git reset --hard <commit>    # Reset to commit (dangerous)

Remote Operations

git remote -v                # Show remotes
git remote add origin <url>  # Add remote
git fetch                    # Download remote changes
git push origin <branch>     # Push branch to remote
git pull origin <branch>     # Pull branch from remote

20. Common Mistakes to Avoid

  • Committing directly to main branch
  • Not pulling before starting work
  • Making huge commits with many unrelated changes
  • Using vague commit messages like "fix" or "update"
  • Forgetting to push your changes
  • Not communicating with team about conflicts
  • Committing sensitive information (passwords, API keys)
